# SPEC.md — Slim-yan（ダイエット総合サポートアプリ）

## 1. Objective（目的・対象ユーザー）

「痩せたい、でも記録がめんどくさい」を解決する Android ネイティブアプリ。

### コアバリュー
- **AIが相談パートナー**：食事プラン・筋トレメニュー・リカバリ案を Claude が生成し、**アプリのデータにそのまま落とし込める**（相談 → 即実行 → 記録 → 翌日も継続のループ）
- **入力コストを最小化**：よく食べるものは登録しておいてタップ一発で記録
- **食事プランを事前に組む**：1日の食事を朝のうちに決めて、その通りに動く
- **崩れたら即リカバリ**：飲み会・食べ損ね・過食が起きた瞬間に当日〜翌日の修正プランを Claude API が提案

### このアプリの差別化ポイント
単に Claude とチャットするだけなら本家 Claude アプリで足りる。本アプリの価値は **「AIが生成した plan が、確認ひとつでアプリの食事枠・筋トレ記録になる」** こと。生成 → プレビュー → 確認保存 → 実行・記録 が1本のループでつながる。

### 解決する課題
- 何を食べ／鍛えればいいか自分で組めない → AI が目標に沿ったプランを生成し、確認して保存
- カロリーが分からない → マイ食品登録 ＋ AI 栄養推定で解決
- 予定が崩れたときどうすれば良いか分からない → AI が当日の残り食事・翌日プランを提案
- 進捗が体感でしか分からない → 体重・カロリーグラフで可視化

### 対象ユーザー
現状は**自分専用**（個人利用・ログイン/アカウントなし）。ただし**将来の公開を見据える**。

> **黄金ルール：個人データはコードに焼き込まない。**
> 機能（テンプレ・買い物リスト・監査・リマインド等）はすべて汎用で作る。個人の常用食品・週間テンプレ・目標値は、シードやロジックに埋めず**アプリのUIからユーザーがDBに入れる**。
> こうすれば「自分向けフル最適化（自分の端末のデータ）」と「公開時の汎用性（コードは個人色ゼロ）」が両立し、公開判断時の手戻りがゼロになる。

## 2. 機能スコープ（MVP）

### ① ユーザープロフィール・目標設定
- 身長・体重・年齢・性別・活動量を入力
- TDEE（総消費カロリー）を自動計算
- 目標体重・達成期日を設定 → 1日の摂取カロリー目標を自動算出
- 目標PFC（タンパク質・脂質・炭水化物）も設定可

### ② 体重記録
- 毎日の体重を記録
- グラフで推移表示（週・月単位）
- 目標体重への残りを表示

### ③ マイ食品登録
- 食品名・一食分のカロリー・PFC・一食の量(g)を登録（よく食べる常用食品＝週間テンプレの部品）
- よく食べるものをすぐ呼び出せる・お気に入り
- AI栄養推定（食品名 → 一食分のkcal/PFC/標準量を推定、確認して保存）

### ④ 週間メニューテンプレート（★本アプリの肝）
- **曜日 × 食事枠（朝/昼/夜/間食）× 品目** で定型メニューを固定する
- 各品目はマイ食品から選ぶ（量はグラムで指定、kcal/PFCは自動計算）
- 一度固めたら「毎週これを回す」運用。曜日タイプ（在宅／出社／自由日）で内容を変えられる
- テンプレ → 当日の食事記録へ**ワンタップ展開**（今日の曜日の品目を `MealEntry` に流し込む）
- 週合計・曜日別のカロリー/PFC を表示（目標との差分も）

### ⑤ 買い物リスト（テンプレから自動算出）
- 週間テンプレを集計し、**各食材が1週間で何個／何g必要か**を自動算出（例：サラダチキン×13）
- マイ食品の一食量を単位に個数換算
- 任意で「N週間分」をまとめ買い用に倍掛け（MVPは1週間）
- ※これはローカル計算のみ。AI不要

### ⑥ リマインド通知（食事＋筋トレ）
- **食事**：枠ごと（朝/昼/夜/間食）に通知時刻を設定
- **筋トレ**：トレ日の通知時刻を設定（例：月・水・金 19:00）
- いずれも曜日指定可（出社日だけ／トレ日だけ等）。ON/OFF切替
- 端末ローカル通知（WorkManager）。Android 13+ は通知権限を要求
- リマインドは汎用 `Reminder`（type=meal/workout）で一元管理

### ⑦ AI食事プラン監査（Claude API → §7.1）
- 「今日の分の生成」ではなく、**固めた週間テンプレを Claude に監査させる**のが主役
- 診断内容：カロリー赤字が安定して出てるか／タンパク質は足りてるか・盛りすぎてないか／塩分・食物繊維・コストの盲点
- 数字を洗い直した**修正提案（確定版）**を返す → 確認してテンプレに反映
- 微調整はチャットで（「もっとタンパク多め」「土曜の昼だけ自由に」等）

### ⑧ リカバリ提案（Claude API → §7.2 / 実装済みT8）
- プランから外れたとき（過食・食べ損ね・急な外食）にトリガー
- 当日の残り食事案 + 翌日プラン修正案をカード表示。採用した残り食事案は確認 → 食事枠に反映可能

### ⑨ 筋トレ記録（固定プログラム＋進捗追跡）
- 週3・全身6種目の**固定プログラム**を持つ（種目 × セット×レップ × 重量）
- セット記録：種目 × 重量 × レップ × セット。前回値を初期表示
- **ダブルプログレッション**支援：3セット全部で上限回数に到達したら「次回は重量UP」と提案
- 部位別・種目別の履歴
- トレ日のリマインド通知（→ ⑥ 汎用 Reminder type=workout）
- AI筋トレメニュー生成・種目入替（→ §7.3）：部位・目的・レベルから生成、プレビュー → 確認 → 保存

### ⑩ ホームダッシュボード
- 今日のカロリー摂取状況（残量バー）
- 体重の直近トレンド
- 今日の食事プランのサマリ（テンプレ展開状況）
- 筋トレ実施有無
- リカバリ提案トリガー

### スコープ外（MVP では作らない）
- バーコードスキャンでカロリー取得
- 外部DB（カロリーSlism等）との連携
- 有酸素運動の消費カロリー計算
- 週次レビュー（Claude API活用）← P2 以降
- ソーシャル機能・クラウド同期

## 7. Claude API 連携仕様

### 7.0 連携の基本方針（ハイブリッド）

Claude は本アプリの **相談パートナー**。連携は次の2層で構成する：

1. **構造化生成**：ボタン/フォームから Claude を呼び、JSON で plan を受け取る（食事プラン・筋トレメニュー・リカバリ案）
2. **チャット微調整**：生成された plan に対し「もっとタンパク質多めで」等の指示を出して**再生成**できる（同一生成セッション内の短いマルチターン。永続的な会話履歴は持たない）

**共通フロー**：`生成 → プレビュー（合計カロリー/PFC表示）→（必要なら微調整して再生成）→ ユーザー確認 → アプリのデータに保存`

- **保存方針**：AI出力は必ず**プレビュー → ユーザー確認**を経てから保存する（自動保存しない）。AIの誤りをユーザーが弾ける。
- **API キー管理**：`local.properties` に保存、`.gitignore` で除外。**git に絶対含めない**。
- **Repository 層に閉じ込める**：`AiRepository` で完結させ UI 層は AI 実装を知らない。
- **モデル**：`claude-haiku-4-5`（軽量・低コスト）。応答はコードフェンス除去後に JSON パース。
- **コスト方針**：自由放談チャットは持たない（マルチターンは生成セッション内に限定）ことでトークン消費を予測可能に保つ。将来のサブスク課金の原価設計に直結。

### 7.1 AI食事プラン監査（優先実装＝P0）

「今日の分をゼロ生成」ではなく、**ユーザーが固めた週間テンプレを Claude に監査・修正させる**のが主役（実ワークフローに基づく）。

- **トリガー**：週間テンプレ画面の「AIに診断してもらう」ボタン
- **送るコンテキスト**：週間テンプレ全体（曜日×枠×品目×kcal/PFC）・目標カロリー・目標PFC・TDEE・任意の自由入力（制約・予算・好み）
- **診断観点**：
  - カロリー赤字が**週を通して安定して**出てるか（特定曜日だけ／自由日で帳消しになってないか）
  - タンパク質は週平均で足りてるか・盛りすぎてないか
  - 塩分・食物繊維・コストの盲点
- **受け取るもの**：①診断サマリ（問題点を優先度順）②数字を洗い直した**修正提案**（曜日×枠の差し替え案）
- **プレビュー**：修正前後の週合計・曜日別 kcal/PFC を並べて表示
- **微調整**：チャット欄で指示（「土曜の昼だけ自由に」「プロテイン減らして」）→ テンプレを文脈に再診断
- **保存**：確認すると修正提案を `MealTemplateItem` に反映（差分適用）。**当日への展開は §4 のワンタップ展開で別途**

> 補足：単発の「今日だけプラン生成」は §7.2 リカバリ提案でカバー。日々の主役はテンプレ＋展開。

### 7.2 リカバリ提案（実装済み T8 / 強化はP1）

- **トリガー**：ホームの「AIリカバリ提案を取得」ボタン（目標未設定時は設定導線ボタンに切替）
- **送るコンテキスト**：今日の目標カロリー・摂取実績・残り・目標PFC
- **受け取るもの**：当日の残り食事案 + 翌日プラン修正案 + 一言メッセージ（カード表示）
- **強化（P1）**：当日の残り食事案を §7.1 と同じ MealEntry 保存フローに乗せる

### 7.3 AI筋トレメニュー生成（P1）

- **トリガー**：筋トレタブの「AIにメニューを作ってもらう」ボタン
- **送るコンテキスト**：対象部位・目的（増量/減量維持）・レベル・直近の種目履歴
- **受け取るもの**：種目リスト。各種目は `name, muscleGroup, sets[{weight, reps}]`
- **保存時の課題**：`SetEntry.exerciseId` が必須。生成種目名を Exercise マスタと突合し、無ければ新規 Exercise を作成してから SetEntry を保存する
- **保存**：確認すると当日の SetEntry として一括保存

## 10. Boundaries

### Always do
- 端末内 Room に保存、オフラインで基本動作（AI機能のみオンライン必須）
- APIキーを git に含めない
- AI生成物は**プレビュー → ユーザー確認**を経てから保存する
- AI提案は医療アドバイスではなく一般的な提案である旨を明示（免責）
- **個人データはコードに焼き込まない**（シード/ロジックは汎用のみ。個人の食品・テンプレ・目標はUI経由でDBに入れる → §1黄金ルール）

### Ask first
- 新しい外部ライブラリの追加
- データモデルの破壊的変更
- MVP スコープ外機能
- 自由放談型チャット（永続会話履歴）の導入 ← コスト構造が変わるため

### Never do
- アカウント・クラウド同期
- ネットワーク通信（Claude API以外）
- 個人データの外部送信（Claude APIへのリクエストは除く）
- AI出力をユーザー確認なしに自動保存

### 通知（食事リマインド）の方針
- 端末ローカル通知のみ（WorkManager）。サーバープッシュは使わない
- Android 13+ は `POST_NOTIFICATIONS` を実行時要求。拒否されても他機能は動く
- 厳密な時刻保証が必要なら `SCHEDULE_EXACT_ALARM` を検討（まずは WorkManager の概ねの時刻で十分）
