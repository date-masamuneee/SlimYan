# 実装計画 — Slim-yan

## 依存グラフ

```
[P0: プロジェクト雛形]
         │
         ▼
[P1: データ層 (Room + DI)]
  Entity / DAO / DB / Repository
         │
         ├──────────┬──────────┬──────────┐
         ▼          ▼          ▼          ▼
    [T3 設定]  [T4 体重]  [T5 食品]  [T7 筋トレ]
   UserProfile BodyWeight  Food CRUD  Exercise/Set
         │          │          │
         ▼          │          ▼
    [TDEE計算]      │    [T6 食事プラン・記録]
                    │       MealEntry
                    │          │
                    └────┬─────┘
                         ▼
                   [T8 ダッシュボード]
                         │
                         ▼
                   [T9 リカバリ提案]
                    Claude API 連携
```

- **データ層**が全UIの土台。ここが通らないと記録系すべて死ぬ。
- **設定・体重・食品・筋トレ**は互いに独立した縦パスなのでデータ層通過後に並行作業可能。
- **食事プラン**は食品マスタに依存（食品がないと選べない）。
- **ダッシュボード**は体重・食事・筋トレすべての読み込みに依存。
- **リカバリ提案**はダッシュボードのデータ（今日の実績）に依存。

## スライス方針（縦切り）

横（全DAO→全VM→全画面）ではなく **1機能=1本の通し経路**で検証する。

1. 設定パス：プロフィール入力 → TDEE自動計算 → 保存 → 再起動後も残る
2. 体重パス：体重入力 → 保存 → グラフに反映
3. 食品パス：食品追加 → 一覧表示 → 削除
4. 食事パス：食品選択 → プラン追加 → チェック → カロリー残量反映
5. 筋トレパス：種目選択 → セット追加 → 当日一覧反映
6. ダッシュボードパス：各記録 → サマリに正しく集計される
7. リカバリパス：ボタン押下 → API呼び出し → カード表示

## リスクと対処

| リスク | 対処 |
|--------|------|
| Ktor Client の設定ミス | P0 でビルドを通す段階で依存解決を確認 |
| Claude API のレスポンス形式ズレ | JSONスキーマを固定し、パース失敗時はエラーカードを表示 |
| TDEE計算の精度 | Harris-Benedict式を採用、計算ロジックはユニットテストで担保 |
| MealEntry のカロリーキャッシュ整合性 | Food を編集しても過去の MealEntry は変えない（記録時点の値を保持） |
